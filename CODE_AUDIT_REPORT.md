## 📋 ERROR REPORT

### 🔴 CRITICAL

---

**[CRITICAL] `app.py` — Нет транзакционной защиты в `create_operation` и `update_operation`**
Расположение: `app.py:~630–680`, `~700–770`

**Проблема:** Операция создания записывает в несколько таблиц (`operations`, `operation_items`, `lots`, `lot_consumptions`) и обновляет `products.current_stock` без явной транзакции с обработкой ошибок. При исключении на середине цикла (например, `fifo_consume` упал) БД остаётся в частично изменённом состоянии.

**Импакт:** Повреждение остатков — `current_stock` уедет, а `lot_consumptions` не запишется. Данные склада становятся недостоверными.

**Действие:** Обернуть весь блок создания/изменения операции в `try/except` с `db.rollback()` при исключении.

---

**[CRITICAL] `app.py` — `create_operation` не валидирует `op_type`**
Расположение: `app.py:~611`

**Проблема:** `op_type = request.form.get("type", "IN")` принимает любую строку из формы. БД имеет CHECK `IN/OUT/ADJUST`, но в коде нет проверки до INSERT. При неожиданном значении произойдёт `sqlite3.IntegrityError` без обработки.

**Импакт:** Необработанное исключение → 500 для пользователя; при отключённом CHECK — запись мусора в БД.

**Действие:** Добавить `if op_type not in ("IN", "OUT", "ADJUST"): abort(400)` перед INSERT.


---

**[CRITICAL] `app.py` — Двойное применение корректировок инвентаризации**
Расположение: `app.py:~1020–1060` и `app.py:~1080–1120`

**Проблема:** При POST с `save_action="complete"` метод `inventory_session` сам применяет корректировки (`INSERT INTO operations`, `UPDATE products`). Но также существует отдельный роут `inventory_complete`, который делает то же самое. Если клиентский JS или форма вызовут оба — корректировки применятся дважды.

**Импакт:** Двойное изменение остатков, дублирование ADJUST-операций, порча финансовой статистики.

**Действие:** Убрать логику завершения из `inventory_session` и редиректить на `inventory_complete`, либо полностью удалить роут `inventory_complete` как дублирующий.

---

**[CRITICAL] `app.py` — `fifo_remove_lot` не восстанавливает `lot_consumptions`**
Расположение: `app.py:~530–540`

**Проблема:** При удалении IN-операции `fifo_remove_lot` обнуляет `remaining_qty` у лота если `consumed > 0`, но не удаляет и не корректирует записи `lot_consumptions`, ссылающиеся на этот лот. При последующих FIFO-расчётах старые потребления "висят" в таблице.

**Импакт:** Некорректный расчёт себестоимости FIFO, потенциальные phantom consumptions.

**Действие:** При `consumed > 0` также очищать или корректировать связанные `lot_consumptions`.

### 🟠 HIGH

---

**[HIGH] `app.py` — `stock_print` импортирует `datetime` внутри функции, теряя `timezone`**
Расположение: `app.py:~1150`

**Проблема:** `from datetime import datetime` внутри `stock_print` перекрывает уже импортированный `datetime` с `timezone`. Используется `datetime.now()` без `timezone.utc` — время берётся локальное серверное, не UTC.

**Импакт:** Время на печатной форме будет расходиться с остальным приложением при работе на сервере с не-UTC временной зоной.

**Действие:** Удалить внутренний импорт, использовать уже доступный `datetime` и `timezone.utc`.


---

**[HIGH] `app.py` — `stats_detail`: хрупкая текстовая замена строк в SQL**
Расположение: `app.py:~840–865`

**Проблема:** Конструкция `prod_filter.replace('AND oi.product_id','AND p.id').replace(...)` — хрупкая текстовая замена. Параметры `prod_params` передаются трижды, что легко нарушить при изменении запроса.

**Импакт:** При рефакторинге вероятен неправильный порядок параметров → некорректные результаты или `sqlite3.ProgrammingError`.

**Действие:** Переписать запрос без текстовой замены, явно указав нужные условия.

---

**[HIGH] `app.py` — Непоследовательная политика паролей**
Расположение: `app.py:~420`

**Проблема:** Роут `change_password` принимает пароль от 4 символов без проверки на заглавные/строчные буквы и спецсимволы. Роут `setup` устанавливает строгие требования (12+ символов, mixed case, цифры, спецсимволы).

**Импакт:** Администратор может ослабить пароль через смену. Непоследовательная политика безопасности.

**Действие:** Привести валидацию `change_password` к тем же требованиям, что и в `setup`.

---

**[HIGH] `app.py` — `settings` action `timezone` не валидирует значение**
Расположение: `app.py:~320`

**Проблема:** `tz = request.form.get('timezone', 'UTC')` сохраняется без проверки. Пользователь может ввести произвольную строку, которая не является валидной IANA-зоной.

**Импакт:** `ZoneInfoNotFoundError` при следующем рендеринге страниц с датами; сбой всего UI.

**Действие:** Проверять `tz` по списку `COMMON_TIMEZONES` или пытаться создать `ZoneInfo(tz)` перед сохранением.

---

**[HIGH] `app.py` — OUT-операция допускает отрицательный остаток**
Расположение: `app.py:~660`, `~760`

**Проблема:** OUT-операция не проверяет `current_stock >= qty` перед `UPDATE products SET current_stock = current_stock - ?`.

**Импакт:** Отрицательные остатки на складе, некорректный FIFO (`fifo_consume` вернёт `None`).

**Действие:** Добавить проверку остатка перед OUT и возвращать понятную ошибку пользователю.

### 🟡 MEDIUM

---

**[MEDIUM] `app.py` — `inject_globals` открывает новые соединения с БД на каждый запрос**
Расположение: `app.py:~240–248`

**Проблема:** `get_setting` внутри `inject_globals` использует `sqlite3.connect(DB_PATH)` (не `get_db()`), открывая отдельное соединение для каждого из 3–4 вызовов при каждом рендеринге шаблона.

**Импакт:** Избыточные открытия соединений с БД, снижение производительности при нагрузке.

**Действие:** Переписать `inject_globals` для использования `get_db()` из контекста запроса.

---

**[MEDIUM] `app.py` — `reorder_products` не валидирует id продуктов**
Расположение: `app.py:~590`

**Проблема:** JSON-массив `order` принимается без валидации. Можно передать несуществующие id.

**Импакт:** Тихая ошибка; потенциально можно использовать для зондирования БД.

**Действие:** Валидировать, что все id из `order` существуют в таблице `products`.

---

**[MEDIUM] `app.py` — Публичный доступ к `uploaded_file` без ограничений**
Расположение: `app.py:~256`

**Проблема:** Роут `/uploads/<path:filename>` в списке `PUBLIC` доступен без аутентификации. Если в `UPLOAD_FOLDER` попадут другие файлы (через баг загрузки), они будут публично доступны.

**Импакт:** Нежелательное раскрытие файлов.

**Действие:** Ограничить доступ только к известным именам файлов (например, только `org_logo.*`) или задокументировать намеренность.

---

**[MEDIUM] `app.py` — `local_dt_filter` делает SELECT на каждое отображение даты**
Расположение: `app.py:~220–232`

**Проблема:** Фильтр `localdt` запрашивает `get_setting('timezone')` при каждом вызове. На странице истории может быть 20+ записей с датами → 20+ дополнительных SELECT.

**Импакт:** Значительное количество лишних запросов к БД на рендер.

**Действие:** Кэшировать значение timezone в `g` или передавать его в шаблон один раз.

---

**[MEDIUM] `app.py` — `inventory_complete` не проверяет права: любой завершит чужую инвентаризацию**
Расположение: `app.py:~1080`

**Проблема:** Роут не проверяет, что `session['user_id'] == current_user.id` или `current_user.role == 'admin'`.

**Импакт:** Обычный пользователь может применить корректировки чужой незавершённой инвентаризации.

**Действие:** Добавить `if session['user_id'] != current_user.id and current_user.role != 'admin': abort(403)`.

---

**[MEDIUM] `app.py` — `init_db` не закрывает соединение при исключении**
Расположение: `app.py:~120`

**Проблема:** `db.close()` только в конце функции. При исключении во время миграции соединение не будет закрыто.

**Импакт:** Утечка дескриптора файла при старте приложения.

**Действие:** Обернуть тело `init_db` в `try/finally: db.close()`.

### 🔵 LOW

**[LOW] `app.py` — Лишний импорт `datetime` внутри `calc_danger_thresholds`**
Расположение: `app.py:~84`

**Проблема:** `from datetime import datetime, timedelta` внутри функции — избыточный импорт, модули уже доступны на уровне файла.

**Действие:** Удалить внутренний импорт.

---

**[LOW] `app.py` — `debug=True` захардкожен в `__main__`**
Расположение: `app.py:~1389`

**Проблема:** При случайном запуске `python app.py` в продакшене включится Werkzeug debugger.

**Действие:** Читать флаг из окружения: `debug=os.environ.get('FLASK_DEBUG', '0') == '1'`.

---

**[LOW] `app.py` — Отсутствует индекс на `lots(product_id, remaining_qty)`**

**Проблема:** FIFO-запрос в `fifo_consume` делает `WHERE product_id=? AND remaining_qty > 0`. Нет индекса по `remaining_qty`.

**Импакт:** При большом количестве лотов — полное сканирование таблицы.

**Действие:** Добавить `CREATE INDEX IF NOT EXISTS idx_lots_product_remaining ON lots(product_id, remaining_qty)` в `init_db`.

---

**[LOW] `app.py` — `history` принимает `page` без валидации минимума**
Расположение: `app.py:~900`

**Проблема:** `page = int(request.args.get("page", 1))` без `try/except` и без проверки `page >= 1`. При `?page=0` offset станет отрицательным.

**Действие:** Добавить `page = max(1, page)` и обернуть в `try/except ValueError`.
